# 新项目协作机制总览

状态：`candidate_for_review`，未激活。

## 目标

为“大米低 GI 跨境”建立一个最小协作操作系统：ChatGPT 负责判断、收束和下发；Codex 负责在授权边界内读取、落库、验证和回报；用户保留业务、合规、资金和外部沟通的最终决定权。

## 四层架构

### 1. 静态协作规则层

长期稳定，只记录角色、任务结构、事实标签、权限、阻断和完成真实性。候选入口是 `04_ChatGPT_Project项目指令_候选版.md` 与 `05_AGENTS候选版_仅供评审.md`。

### 2. 动态项目事实层

只记录经证据和授权确认的产品、市场、合规、渠道、实验、风险和决策。候选入口是基于 `06_当前正式事实模板.md` 另行建立的正式文件。本轮模板为空，不把既有方案自动升级为事实。

### 3. 任务执行层

每轮用独立任务卡串联需求对齐、实现设计、影响面、允许/禁止范围、执行、验证和回报。候选入口是 `08`、`10`、`11`。

### 4. 证据与历史层

保存研究来源、原始凭证、决策依据、实验结果、失败记录、历史状态和被淘汰方案。`latest` 只做当前入口，历史记录不可被覆盖；原始证据与正式替代关系必须可追溯。

## 角色边界

- ChatGPT：识别真实问题、判断阶段、区分事实与假设、生成完整执行单；不得把讨论写成正式决策。
- Codex：回读事实源、检查影响面、执行获准的本地任务、验证与如实回报；不得替用户作业务承诺或越级推进状态。
- 用户：确认市场、合规声明、报价、合同、付款、样品、外发、预算、删除和规模化。
- GitHub：未来若启用，承载版本化规则、动态事实与证据，不自动等同于业务通过。

## 最小闭环

`用户意图 -> ChatGPT 需求对齐 -> 实现设计与执行单 -> Codex 影响面检查 -> 授权范围内执行 -> 验证与状态证据 -> 用户人工闸门 -> 正式事实更新`

## 状态真实性

任何结果至少区分：`technical_validation`、`evidence_validation`、`human_review`、`business_decision`、`external_sync`。低层通过不能自动推导高层通过；文件生成不等于机制激活，上传包生成不等于 ChatGPT Project 已同步。

## 当前裁决

- 协作底座：候选版已设计，等待用户审核。
- 动态事实：待人工确认，不能从既有业务方案自动导入。
- 机制激活：未发生。
- 业务执行：未开始。
