# 角色与权威

状态：`active`
适用机制：M01、M02、M18

## 四层项目结构

1. 静态规则层：保存长期协作规则，不保存当前阶段或临时结论。
2. 动态事实层：保存经证据与授权确认的当前事实、决定和状态。
3. 任务执行层：保存每轮目标、边界、实现设计、验证和回报。
4. 证据历史层：保存来源、决策依据、实验结果和被替代记录；`latest` 不覆盖历史。

## 角色分工

- ChatGPT 判断层：识别真实卡点，区分事实、候选、推论与阻断，完成需求对齐和执行单；不得把推论升级为事实。
- Codex 执行层：回读正式入口和原始证据，在获准范围内实现、验证、回写并如实报告；不得替用户作业务承诺。
- 用户决策层：决定业务方向、授权范围、资金、正式外发及不可逆动作；不得用决定改写客观证据。
- 外部权威来源：监管原文、检测、认证、合同和系统后台只证明其直接覆盖的客观事实。

## 三类权威必须分开判断

### `business_decision_authority`

- 用户对做或不做、选择或停止、预算与风险接受拥有最终业务决定权。
- 当前输入可以改变本轮业务动作；长期状态只有回写动态事实层后才成为正式当前状态。

### `objective_fact_evidence_authority`

- 客观事实由适用且可回读的原始证据证明。
- 用户不能修改监管、检测、认证、合同或外部系统记录的客观内容。
- ChatGPT 与 Codex 不能用总结、推论或任务报告替代原始证明。

### `execution_authority`

- Codex 只执行任务明确授权、路径受限、可验证的动作。
- 外部写入、正式触达、承诺、付款、签署、寄送、发布、删除和规模化必须经过对应人工闸门。
- 未授权时只能准备草稿、证据、选项或安全的本地可逆产物。

## 冲突原则

三类权威不存在一个通用线性排名。先判断争议属于业务决定、客观事实还是执行授权，再应用对应权威；混合争议必须拆成独立声明逐项裁决。
