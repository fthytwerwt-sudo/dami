# 工程深度、执行预算与协作有效性

状态：`active_with_deferred_component`
适用机制：M19、M20、M21

## M19 当前裁决

`M19 activation_status: DEFERRED`

当前只保留按风险和重复度选择工程深度的判断思想，不建立复杂 schema、图编排、多节点运行时或重型自动化。文档规则能解决时不建立状态机代码；真实重复任务未出现时不建立 L2/L3 工程系统。

重新评审条件：出现稳定重复流程、人工成本或错误率有连续证据、自动化边界可测试、维护责任和回滚方案明确，并经用户授权。满足条件只代表可以评审，不代表自动启用。

## 执行预算

每个任务选择能证明目标的最轻方案，并明确：

- 任务风险与可逆性。
- 预计读取、写入、工具调用和外部成本。
- 最小验证集与停止条件。
- 何时升级为脚本、自动化或更深工程。
- 新增机制的维护者、收益证据和退出方式。

不得因“机制完整”而增加文件、脚本、依赖或运行层。已有规则、模板或简单检查可满足时优先复用。

## 第一次复盘

第 3 个真实任务结束后进行第一次协作机制复盘。复盘至少检查：

- 是否减少重复解释。
- 是否减少事实误升级。
- 是否减少漏项。
- 是否减少未授权外部动作。
- 是否增加过多维护成本。
- 哪些字段长期没有价值。
- 哪些阻断过于频繁。
- 哪些机制应简化、合并或自动化。

复盘必须基于三次真实任务证据，不以文件数量或形式完整作为成功标准。复盘建议经批准后再修改正式规则；未批准建议只进入候选记录。
