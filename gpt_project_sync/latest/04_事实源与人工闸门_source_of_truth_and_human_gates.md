# 角色与权威

状态：`active`
适用机制：M01、M02、M18

## 四层项目结构

1. 静态规则层：保存长期协作规则，不保存当前阶段或临时结论。
2. 动态事实层：保存经证据与授权确认的当前事实、决定和状态。
3. 任务执行层：保存每轮目标、边界、实现设计、验证和回报。
4. 证据历史层：保存来源、决策依据、实验结果和被替代记录；`latest` 不覆盖历史。

## 角色分工

- ChatGPT 判断层：识别真实卡点，区分事实、候选、推论与阻断，完成需求对齐和执行单；不得把推论升级为事实。
- Codex 执行层：回读正式入口和原始证据，在获准范围内实现、验证、回写并如实报告；不得替用户作业务承诺。
- 用户决策层：决定业务方向、授权范围、资金、正式外发及不可逆动作；不得用决定改写客观证据。
- 外部权威来源：监管原文、检测、认证、合同和系统后台只证明其直接覆盖的客观事实。

## 三类权威必须分开判断

### `business_decision_authority`

- 用户对做或不做、选择或停止、预算与风险接受拥有最终业务决定权。
- 当前输入可以改变本轮业务动作；长期状态只有回写动态事实层后才成为正式当前状态。

### `objective_fact_evidence_authority`

- 客观事实由适用且可回读的原始证据证明。
- 用户不能修改监管、检测、认证、合同或外部系统记录的客观内容。
- ChatGPT 与 Codex 不能用总结、推论或任务报告替代原始证明。

### `execution_authority`

- Codex 只执行任务明确授权、路径受限、可验证的动作。
- 外部写入、正式触达、承诺、付款、签署、寄送、发布、删除和规模化必须经过对应人工闸门。
- 未授权时只能准备草稿、证据、选项或安全的本地可逆产物。

## 冲突原则

三类权威不存在一个通用线性排名。先判断争议属于业务决定、客观事实还是执行授权，再应用对应权威；混合争议必须拆成独立声明逐项裁决。


# 事实源与冲突裁决

状态：`active`
适用机制：M03、M16、M28

## 来源适用规则

- 用户本轮明确决定：约束业务选择与授权，不证明客观事实。
- 当前正式事实、决定与状态文件：是项目当前统一入口，每条应能回指证据和确认记录。
- 原始监管、检测、认证、合同和系统记录：对其直接覆盖的客观事实具有最高证明力。
- 当前有效的研究来源：支持候选判断，必须记录日期、范围、限制和复核期限。
- 仓库内容：只证明仓库内实现、规则和记录状态，不证明外部业务或合规状态。
- 最新任务报告：只证明该轮执行和验证，不自动覆盖正式当前文件。
- 历史、计划、摘要、AI 推论和未验证猜测：仅用于追溯或候选判断。

## 冲突裁决流程

1. 将冲突拆成可单独验证的声明，并标记其权威类型。
2. 为每个来源记录类型、签发者、日期、适用范围、有效期、完整性和替代关系。
3. 回读直接原始来源；搜索片段、索引、摘要和 AI 总结只能作为线索。
4. 判断来源是否直接证明该声明；较新日期不自动代表正式替代。
5. 用户决定与客观证据冲突时，分别保留决定和证据，并显式记录风险与限制。
6. 无法裁决时停止状态推进，进入 `human_decision_required` 或具体 `blocked_*` 状态。

## 原文件回读

用于事实确认或业务执行前必须回读原网页、原文件、原始凭证或原系统记录。若只有摘要、无法回读、已过复核期或适用范围不明，结论只能标为 `candidate` 或 `unverified`，不得进入正式业务执行。

## 有效期、替代与历史

- 易变化事实必须记录 `review_due` 或有效期。
- 替代事实使用 `superseded_by`、生效时间、批准者和证据，不删除旧记录。
- `latest` 只指向当前入口；历史记录不可重写。
- 存在本地私有来源登记时，Codex 可在授权范围内读取；未完成权威裁决的内容只能作为上下文，不能进入正式事实。


# 执行安全与人工闸门

状态：`active`
适用机制：M17、M18

## 默认执行边界

- 只修改任务明确允许的路径，不纳入无关脏改。
- 不读取、打印、复制或提交密钥；需要秘密才能继续时使用 `blocked_secret_access_required`。
- 外部动作默认不授权；本地草稿不等于外发批准。
- 先用 `dry-run`、沙箱、最小样本和可逆步骤验证能力；探测结果不得冒充正式执行。
- 删除、覆盖、迁移正式数据或无法可靠回滚时必须停止并请求授权。
- 候选研究、计划和机器验证不得写成正式事实或人工通过。

## 必须人工确认

- 正式市场、渠道、商业模式和规模化选择。
- 产品合规、健康、营养或医疗相关声明。
- 正式报价、交易数量、交付条件、代理条件及客户承诺。
- 合同、法律文本、付款、采购、付费资源和预算。
- 样品寄送、地址或个人信息使用、物流承诺。
- 正式邮件、消息、批量触达、广告、发布、上架和关键账号操作。
- 覆盖正式事实、改变正式业务状态、删除数据或不可逆动作。
- 需要法律、监管、税务、医疗或其他专业判断的最终结论。

授权记录最低字段：`action`、`target`、`scope`、`limit`、`valid_from`、`valid_until`、`approved_by`、`revocation`、`evidence`。

## 公开仓库安全

```yaml
business_sensitive_git_write_blocked_while_repo_public: true
```

仓库为公开状态时，不得写入客户、供应商、成本、报价、合同、认证或检测原件、个人信息、秘密及其他商业敏感数据。不得擅自改变仓库可见性。敏感资料只能留在已批准且被 Git 排除的私有位置。

## 具体阻断

- `blocked_authorization_required`
- `blocked_compliance_evidence_missing`
- `blocked_write_scope_violation`
- `blocked_secret_access_required`
- `blocked_repo_visibility_for_sensitive_data`
- `blocked_capability_unverified`


# 供料与执行中补料（supply and mid-task supply）

状态：`active`
适用机制：M16、M22
适用习惯：H12-H14

## 来源裁决

顺序是：当前正式仓库原文件 → 经授权私有来源 → 外部权威原始来源 → 搜索/AI 摘要导航 → 条件第二意见。来源冲突、低置信度、高风险、用户明确要求时可触发只读第二意见；不绑定固定供应商，第二意见不能替代原文件。

每次记录 `retrieval_manifest`、`source_readback_status`、`source_conflicts`、`external_research_needed`、`second_opinion_trigger`。检索结果、聊天记忆、静态 GPT 包或摘要没有原文件回读时只能是线索。

## pre_task_supply_pack 与 mid_task_supply_pack

执行前供料包至少包含：`task_goal`、`current_state`、`must_read_sources`、`exact_relevant_snippets`、`constraints`、`conflicts`、`missing_information`、`decision_authority`、`allowed_actions`、`forbidden_actions`、`next_input_for_codex`。每个来源要有 `source_path`、`readback_status` 与关键原文片段。

子任务缺上下文、验证失败、高风险写入或冲突未解时生成执行中补料包。`continue_allowed: false` 时禁止继续写。无法形成可回读资料链时进入 `supply_repair`，而不是猜测。

## 安全

不读取、打印或提交 `.env`、secret 或 key；不调用付费 API；高风险能力先 dry-run、只读探测或 Fixture。技术可运行不等于业务可用。
